# Getting Started

Everything you'll touch when adopting this template for a new project.
Work top-down — the order is roughly "biggest visible change first."

## What this template is

A pnpm workspace with two packages:

- **`packages/docs/`** — the documentation site. Vike (SSR + client
  routing), MDX, Tailwind v4, Shiki, a search index built at compile
  time. This is what gets deployed.
- **`packages/library/`** — a placeholder package containing a single
  example component (`<Button />`). Replace it with your real library,
  rename it, or delete it entirely (see step 3).

The docs site imports from the library via a Vite alias
(`vite.config.ts`), so there's no build step between the two — edit a
library file and the docs hot-reload.

## Install and run

```bash file="terminal"
pnpm install
pnpm dev
```

The dev server boots on `http://localhost:3010` (or the next free port
— `strictPort: false` in `vite.config.ts`). Edit any `+Page.mdx` and
the page hot-reloads.

```bash file="terminal"
pnpm build      # static site under packages/docs/dist/client/
pnpm preview    # serve the built site locally
```

## 1. Site config

Everything visible in the topbar, breadcrumbs, browser tab, and GitHub
link comes from one file. Start here:

```ts file="packages/docs/site.config.ts"
export const siteConfig = {
  brand: 'MyLib',
  subtitle: 'docs',
  repoUrl: 'https://github.com/you/my-lib',
  breadcrumbRoot: 'Docs',
}
```

- `brand` — top-left wordmark **and** the suffix appended to every page
  `<title>` (`Getting Started — MyLib`).
- `subtitle` — small muted text after `brand /`. Set to `''` to hide.
- `repoUrl` — drives the GitHub icon top-right. Empty string hides the
  icon.
- `breadcrumbRoot` — leftmost crumb in both breadcrumbs (page-top and
  topbar). Set to `''` to omit.

## 2. Write pages

Drop a folder under `packages/docs/pages/` with a `+Page.mdx` inside.
Frontmatter drives the sidebar:

```mdx
---
title: Installation
section: Getting Started
order: 0
description: Install the package and run the dev server.
draft: true
---

# Installation

…
```

- `title` — sidebar label, breadcrumb crumb, and the prefix in the
  browser `<title>`.
- `section` — sidebar group label. Pages without one float to the top
  of the sidebar.
- `order` — sort within a section. Leave gaps (0, 10, 20…) so you can
  insert without renumbering later.
- `description` — emitted as `<meta name="description">` for SEO.
- `draft` — optional. Set to `true` while the page is still awaiting
  review; the sidebar entry picks up a **DRAFT** chip until the flag
  is removed or set back to `false`. The page itself still builds,
  renders, and is searchable — the chip is purely a workflow signal
  for reviewers. See **[Drafts](/drafts)** for the full walk-through.

URLs are derived from the folder path:
`pages/installation/+Page.mdx` → `/installation`. Nest folders to nest
URLs (`pages/api/auth/+Page.mdx` → `/api/auth`).

See **[Components](/components)** for the full list of MDX components
and markdown features available in the body.

## 3. The `library/` package — keep, rename, or delete

Three paths. Pick the one that matches your project.

### Option A — Pure docs site, no library

If you're writing standalone docs (a guide, a spec, a handbook) and not
documenting a library that lives in this repo, delete the whole
`library/` package:

```bash file="terminal"
rm -rf packages/library
rm -rf packages/docs/examples
rm -rf packages/docs/pages/library
```

Strip the alias from `vite.config.ts`:

```ts file="packages/docs/vite.config.ts"
// Delete the resolve.alias block entirely:
resolve: {
  alias: {
    library: new URL('../library/src/index.ts', import.meta.url).pathname,
  },
},
```

Drop the `workspace:*` dependency:

```json file="packages/docs/package.json"
// Remove from "dependencies":
"library": "workspace:*",
```

Simplify the root build script:

```json file="package.json"
"build": "pnpm --filter docs build"
```

Run `pnpm install` to refresh the lockfile.

### Option B — Rename to your real package

If this repo also contains the library you're documenting, rename it.
Three `package.json` files touch the name. Pick a name (we'll use
`my-lib` below):

```json file="packages/library/package.json"
{
  "name": "my-lib",
  "version": "0.1.0",
  "description": "What my-lib does"
}
```

```json file="packages/docs/package.json"
{
  "dependencies": {
    "my-lib": "workspace:*"
  }
}
```

Rename the Vite alias key so `import { Foo } from 'my-lib'` resolves to
the workspace source:

```ts file="packages/docs/vite.config.ts"
resolve: {
  alias: {
    'my-lib': new URL('../library/src/index.ts', import.meta.url).pathname,
  },
},
```

Renaming the **folder** (`packages/library/` → `packages/my-lib/`) is
optional and cosmetic — `pnpm-workspace.yaml`'s `packages/*` glob
picks any folder name. If you do rename it, update the relative path
in the alias above.

Update the example imports in `packages/docs/examples/` and the body
of `pages/library/button/+Page.mdx`:

```bash file="terminal"
# Search and replace `from 'library'` → `from 'my-lib'`
grep -rl "from 'library'" packages/docs
```

Run `pnpm install` to refresh symlinks. The
**[Library / Button](/library/button)** page in this template is the
working example of what one of your library pages will look like —
folder layout, imports, Preview wiring — copy that pattern for each
new component.

### Option C — Keep the placeholder

Fine for prototyping. The `library/` package ships with one component
(`<Button />`), one example doc page, and the wiring already in place.
Replace it later when you have a real library to document.

## 4. Release and deploy

The template assumes **Netlify** as the host, wired to three kinds of
branches. Each `pnpm prod` or `pnpm staging` invocation pushes the
current `HEAD` to **two** branches at once — an environment branch and
a versioned snapshot branch.

### `version` in `package.json` is load-bearing

The root `package.json`'s `version` field drives:

- The **`v…` chip** rendered in the topbar (`__APP_VERSION__` injected
  by Vite at build time).
- The **snapshot branch name** every release pushes to: `v/<version>`
  (e.g. `v/0.1.0`, `v/0.2.0-rc.1`).

Bump it **before** running `pnpm prod` / `pnpm staging`. Re-running a
release without bumping force-overwrites the existing `v/<version>`
branch, silently rewriting the deploy preview pinned to that version.

```json file="package.json"
{
  "version": "0.2.0"
}
```

### The release scripts

```json file="package.json"
"scripts": {
  "prod":    "git push -f origin HEAD:netlify-production && pnpm version-branch",
  "staging": "git push -f origin HEAD:netlify-staging && pnpm version-branch",
  "version-branch": "node scripts/push-version-branch.mjs"
}
```

Each script does two pushes:

1. **Force-push HEAD** to a fixed environment branch
   (`netlify-production` or `netlify-staging`) — Netlify watches these
   and rebuilds the live URL on every push.
2. **Force-push HEAD** to a `v/<version>` snapshot branch — Netlify
   serves it as an immutable preview URL like
   `https://v-0-2-0--your-site.netlify.app`.

You get **three URLs per release**: the production URL, the staging
URL, and a permanent versioned URL you can link to from release notes.

### Enable branch deploys in Netlify

Netlify does **not** deploy arbitrary branches by default. Turn it on
once per site:

1. Netlify dashboard → your site → **Site configuration → Build &
   deploy → Branches and deploy contexts → Edit configuration**.
2. Set **Branch deploys** to either:
   - **All** — every pushed branch deploys (simplest).
   - **Let me add individual branches** — explicitly list
     `netlify-production`, `netlify-staging`, and the `v/*` glob.
3. Save.

Without this, `pnpm prod` will push successfully but Netlify will
ignore the branch and nothing will deploy.

<Callout variant="warn" title="Force-push, by design">
  Both `prod` and `staging` use `git push -f`. The environment branches
  (`netlify-production`, `netlify-staging`) and the version branches
  (`v/<version>`) are **deploy targets**, not collaboration branches —
  treat them as overwritable. Land changes via PRs into your default
  branch first, then run `pnpm prod` from there.
</Callout>

### Hosting elsewhere

The build artifact is plain static files under `dist/client/` from
`pnpm build`. Any static host works — Vercel, Cloudflare Pages, GitHub
Pages, S3 + CloudFront, an internal CDN. Replace the `prod` / `staging`
scripts with whatever push or upload command your host expects; keep
`pnpm version-branch` if you still want versioned snapshots.

## Checklist

Before your first release:

- [ ] `site.config.ts` brand / subtitle / repo / breadcrumb set
- [ ] Library renamed, deleted, or knowingly kept as the placeholder
- [ ] If renamed: Vite alias key + `workspace:*` dep + example imports
  all updated
- [ ] Root `package.json` `version` reflects the release you're about
  to push
- [ ] Netlify branch deploys enabled for `netlify-production`,
  `netlify-staging`, and `v/*`
- [ ] `pnpm dev` boots, hot-reload works, no 404s in the sidebar
- [ ] `pnpm build` succeeds and `pnpm preview` looks right
